What is a model?

A model is a function — it turns an input into an output. What makes today's AI models special isn't that they're functions; it's that their rules were grown from data instead of written by a person.

Tier 1 · status: can-apply · first built session 2 · the concept the curriculum had been silently standing on

1.The spine

Strip away everything fancy and a model is a function: input goes in, output comes out. More fully, it's a simplified stand-in for something real that captures the patterns that matter — so you can predict or decide. A map. A weather forecast. A churn model. The model is the rule connecting input to output. Everything else is a detail about where that rule comes from.

✎ In your words “Both the spam filter and Claude are models because they take an input in order to determine or execute an output.” — exactly the spine. Keep this as your anchor.

2.Two independent axes

Two different questions about any model — and they don't have to travel together:

How is it made?  authored rules ⟷ grown from data
How broad is its job?  one narrow task ⟷ general-purpose

A spam filter that learns is {grown, narrow}. An old chess engine was {authored, broad-ish}. Claude is {grown, general}. "It learned from data" and "it can do many things" are two different claims — keep them apart.

✎ In your words You sensed this yourself — "the difference is in how that model is shaped… as well as perhaps its scope." Right on both. This box just gives the two dials names so you can turn them separately.

3.Authored vs grown — the two spam filters

v1 (authored): a programmer writes "if the email says 'free money,' mark it spam." Deterministic, hand-written, human-readable.

v2 (grown): nobody writes the rule. A flexible function with millions of adjustable parameters is shown labeled examples until it reproduces them and handles new ones.

✎ In your words — the baby's brain "Like a baby's brain that has all the preset structures but has to learn what a tree is." This nails the deepest distinction in ML: the architecture (the preset structure, designed by humans) vs the weights (everything learned from experience). Same model, before vs after learning.

And your sharp catch: the programmer's "free money" rule from v1 becomes just one learned pattern among millions in v2 — the model discovers it instead of being told.

4.How the grown model gets its rules

Show it labeled examples → it nudges its knobs to reduce how wrong it is → repeat, millions of times. The "knowledge" ends up stored in the knob-settings (weights), not as readable rules. (The mechanism of the nudging is gradient descent — its own page.)

✎ In your words — you re-derived the loop "It adjusts its knobs and tests against the data… continues to adjust to increase accuracy, learning from how its knob changes impact the output." That is the gradient-descent loop — you rebuilt it from the foundation, unprompted.

5.The question that opens your field

You flagged it with a "(?)": does it test on the same data, or new data? This is the hinge.

↗ Sharpen it — memorization vs generalization Test on the exact emails it trained on and it can score 100% by simply memorizing them — learning nothing transferable (a student who memorized the answer key). The real test is data it has never seen — only that shows it learned the underlying pattern, i.e. that it can generalize. That's why we always hold out a test set. The failure where a model fits its training data but not new data is overfitting.

And this is the doorway to evals (your focus): how do you design a test that proves a model truly understands, rather than cleverly producing what would satisfy the grader?

6.A language model, specifically

A language model is a grown function whose particular job is: given some text, predict what comes next ("The capital of France is ___" → "Paris"). At its core, that's what Claude is — a next-word predictor grown from a vast amount of human text. The startling part: to predict well, the weights are forced to absorb grammar, facts, and reasoning — those capabilities emerge from the prediction task.

✎ In your words "Not being instructed grammar — given a bunch of textbooks to derive the patterns… language in all its forms." Right: structure is learned from text, not taught as rules.
↗ Sharpen it — who writes the labels? For the spam filter, humans label each email. For Claude's core training, the text labels itself: hide the next word, predict it, check against the word that actually came next. That's self-supervised learning, and it's why LLMs can train on essentially all text without armies of labelers.

7.Grown, not written — why your job exists

Because the behavior was grown from data rather than authored, nobody fully knows or directly controls what the model learned. There's an unavoidable gap between "what we trained it to do" and "what it actually internalized." That gap is the reason evals and interpretability and alignment exist.

✎ In your words — the grown child "More like a child that is grown — we can only observe how it behaves and infer how its mind works, instead of looking into the brain." This is precisely the split between evals (study behavior) and interpretability (study internals).
↗ Sharpen it — the beautiful correction Unlike a brain, with a model we can see every single number — every weight, every activation, perfectly, anytime. We have total access a neuroscientist would kill for. And we still can't read meaning off it. So a model isn't a black box because it's hidden — it's a black box because we see everything and understand almost none of it. Interpretability isn't limited by access; it's limited by understanding.

8.One clarification you'll want

"The model" (Claude) is a frozen artifact — a fixed file of billions of weights. Training set those numbers. Inference (chatting) just runs your text through the frozen function. It is not rewiring itself as you talk — within a conversation it reads what's been said as input, but it isn't permanently learning from you.

Key terms (the field's words)

Model
A function mapping input → output; a compressed stand-in for patterns, used to predict or decide.
Parameters / weights
A model's adjustable internal numbers. Their settings are what it knows.
Architecture
The fixed design/structure of the model (human-designed), as opposed to the learned weights.
Machine learning vs rule-based
Rules grown from data vs rules written by a person.
Supervised vs self-supervised
Humans provide labels vs the data supplies its own (e.g. "predict the next word").
Generalization / overfitting
Working on unseen data (the goal) / fitting training data but failing on new data (the trap).
Training vs inference
Setting the weights (once) vs running the frozen model (every use).
Language model / token
A model that predicts the next token (≈ word-piece) of text.

✓Test yourself (later)

In one sentence, what's the difference between an authored model and a grown one?
An authored model runs rules a human wrote; a grown model's rules were discovered automatically by tuning parameters to fit data — its knowledge lives in the weights, not in readable rules.
Why must you test a model on data it has never seen?
Because on its training data it could just be memorizing. Only unseen data shows it learned a transferable pattern (generalization) rather than the answer key.
We can see every weight in a model — so why is it still a "black box"?
Full access ≠ understanding. We can read every number perfectly and still not know what they mean. The barrier is interpretation, not visibility.
What is a language model's core input→output job?
Given the text so far, predict the next token. Grammar, facts, and reasoning emerge as side effects of getting good at that.
